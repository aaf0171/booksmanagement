Oui, tu arrives à un moment où il faut poser la base de la sécurité côté front. Vu ton architecture (Spring Boot API stateless + VueJS), le flux classique est effectivement :

Le frontend envoie login + password
Le backend authentifie

Le frontend utilise ce JWT pour appeler les endpoints protégés

Mais il faut réfléchir dès maintenant au refresh token, car l'ajouter plus tard implique souvent de modifier le modèle de sécurité.

1. Endpoint /auth/token ou /auth/login

Je ferais plutôt un endpoint orienté use case :

Payload :

Réponse :

Le JWT contient par exemple :

{
  "sub": "12345",
  "login": "jc.dusse",
  "roles": [
    "BORROWER"
  ],
  "iat": 1784100000,
  "exp": 1784100900
}

Le sub devrait être l'identifiant technique du login (ta table logins), pas forcément le username.

2. Durée de vie du JWT

Évite de faire un JWT valable plusieurs jours.

Un choix courant :

Access token : 15 minutes
Refresh token : 7 à 30 jours

Pourquoi ?

Parce qu'un JWT est difficile à révoquer.

Si quelqu'un vole un access token valable 7 jours, tu as un problème.

4. Stockage du refresh token côté backend

Je ne mettrais pas le refresh token dans le JWT.

Je créerais une table :

CREATE TABLE refresh_tokens (
    id BIGINT PRIMARY KEY,
    login_id BIGINT NOT NULL,
    token_hash VARCHAR(255) NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    revoked BOOLEAN NOT NULL DEFAULT FALSE,

    CONSTRAINT fk_refresh_login
      FOREIGN KEY(login_id)
      REFERENCES logins(id)
);

Comme pour ton activation token :

tu stockes le hash
tu compares le hash du token reçu
tu peux révoquer

Ça colle bien avec ton approche actuelle.

5. Au logout ?

Avec JWT seul :

DELETE côté navigateur

Le token continue d'être valide jusqu'à expiration.

Avec refresh token :

POST /auth/logout

Le backend fait :

UPDATE refresh_tokens
SET revoked=true
WHERE token_hash = ?

L'access token mourra naturellement après 15 minutes.

Endpoints à prévoir
1. Login
POST /api/auth/login

Entrée :

{
  "login": "user",
  "password": "password"
}

Traitement :

recherche du login dans logins
vérification du mot de passe avec ton PasswordEncoder
vérification que le compte est actif (activation terminée)
chargement des rôles/permissions
génération des tokens

Sortie :

{
  "accessToken": "eyJhbGciOiJIUzI1Ni...",
  "refreshToken": "a9f82c...",
  "expiresIn": 900
}
2. Refresh
POST /api/auth/refresh

Entrée :

{
  "refreshToken": "a9f82c..."
}

Traitement :

hash du refresh token reçu
recherche dans refresh_tokens
vérification :
existe
non expiré
non révoqué
génération d'un nouveau JWT

Sortie :

{
  "accessToken": "nouveau.jwt..."
}
3. Logout (recommandé)
POST /api/auth/logout

Entrée :

{
  "refreshToken": "a9f82c..."
}

Traitement :

UPDATE refresh_tokens
SET revoked = true
WHERE token_hash = ?

Ton LoginService ferait :

public LoginResponse login(LoginCommand command) {

    Login login = loginRepository.findByUsername(command.username());

    passwordEncoder.matches(
        command.password(),
        login.getPassword()
    );

    String accessToken =
        jwtTokenGenerator.generate(login);

    String refreshToken =
        refreshTokenService.create(login);

    return new LoginResponse(
        accessToken,
        refreshToken
    );
}
